# Base vs trained - same model weights, LoRA toggled

**Repository:** <https://github.com/CodeMaverick2/membrane>

Generated by `scripts/hf_jobs/eval_base_vs_trained.py`. The same Qwen2.5 1.5B
weights are evaluated twice: once with the trained LoRA delta switched off
(base), once with it switched on (trained). Each run samples 4 completions
per task using the exact training prompt, then scores each completion through
the live Membrane environment.

- Adapter under test: `continue_deep_seed_5821_steps_1500`
- Samples per task: 4
- Tasks: `dyad_must_refuse_v1` (training task), `dyad_must_comply_v1`,
  `dyad_must_refuse_long_v1`, `triad_must_refuse_v1` (3 transfer tasks)

## Per-task mean reward

| task | base | trained | delta |
|---|---|---|---|
| `dyad_must_refuse_v1` | 0.0000 | 0.9880 | +0.9880 |
| `dyad_must_comply_v1` | 0.0000 | 0.4350 | +0.4350 |
| `dyad_must_refuse_long_v1` | 0.0000 | 0.9957 | +0.9957 |
| `triad_must_refuse_v1` | 0.0000 | 0.6417 | +0.6417 |
| **overall** | **0.0000** | **0.7651** | **+0.7651** |

The base model scores 0 everywhere because it produces 0 % valid Membrane
JSONL - it cannot follow the verb-style action grammar. The trained policy
produces 100 % valid JSONL and 100 % `COMMIT` rate on every task. The lower
score on `dyad_must_comply_v1` is a transfer artefact: the training diet
leans heavily on must-refuse cases, so the trained model occasionally
over-refuses on benign requests.
